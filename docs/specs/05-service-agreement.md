# Service Agreements

**Module:** 05 — Service Agreements
**Status:** Built (Phase 1)
**Entities:** ServiceAgreement, ServiceAgreementMeter

## Overview

The Service Agreement is the core billing unit of the Utility CIS. It is the contract that says: "Account A receives Commodity B at Premise C, measured by Meter(s) D, billed at Rate Schedule E on Billing Cycle F, starting on date G." Every bill generated by the system originates from a ServiceAgreement.

The ServiceAgreementMeter junction table links one or more physical meters to an agreement. Most residential service has one meter per agreement. Commercial and industrial accounts may have multiple meters contributing to a single billing calculation (e.g., a main meter plus a check meter, or multiple sub-meters that are summed for billing).

**Who uses it:** CSRs starting and stopping service, billing staff confirming rate and cycle assignments, field operations staff confirming which agreements a meter belongs to.

**Why it matters:** The ServiceAgreement is the convergence point of all the system's entities. Without a valid ServiceAgreement, meter reads cannot be billed. The status lifecycle enforces orderly service starts and stops, and the meter assignment uniqueness constraint prevents double-billing.

## Entities

### ServiceAgreement

The core billing unit linking account, premise, commodity, rate schedule, billing cycle, and meters.

| Field | Type | Notes |
|-------|------|-------|
| id | UUID | PK |
| utility_id | UUID | Tenant scope |
| agreement_number | VARCHAR(50) | Unique per utility; human-readable identifier |
| account_id | UUID | FK → Account (required) |
| premise_id | UUID | FK → Premise (where service is delivered) |
| commodity_id | UUID | FK → Commodity (what is being measured and billed) |
| rate_schedule_id | UUID | FK → RateSchedule (pricing rules) |
| billing_cycle_id | UUID | FK → BillingCycle (when to read and bill) |
| start_date | DATE | Required; when service begins |
| end_date | DATE | Null = currently active; populated when service ends |
| status | ENUM | `PENDING`, `ACTIVE`, `FINAL`, `CLOSED` |
| read_sequence | INTEGER | Optional; order within a meter reading route |
| created_at | TIMESTAMPTZ | |
| updated_at | TIMESTAMPTZ | |

**Unique constraint:** `[utility_id, agreement_number]`

**Relationships:**
- `account` → Account
- `premise` → Premise
- `commodity` → Commodity
- `rateSchedule` → RateSchedule
- `billingCycle` → BillingCycle
- `meters` → ServiceAgreementMeter[] (one or more meter assignments)
- `meterReads` → MeterRead[] (all reads billed under this agreement)

**Status definitions:**
- `PENDING` — Agreement created but service has not started (e.g., set up in advance of move-in)
- `ACTIVE` — Service is live; reads and billing are occurring
- `FINAL` — Service stop has been initiated; a final read and final bill are being processed
- `CLOSED` — Final bill issued; agreement permanently closed

### ServiceAgreementMeter (Junction)

Links one or more meters to a ServiceAgreement. Tracks when each meter was added to and removed from the agreement.

| Field | Type | Notes |
|-------|------|-------|
| id | UUID | PK |
| utility_id | UUID | Tenant scope |
| service_agreement_id | UUID | FK → ServiceAgreement (required) |
| meter_id | UUID | FK → Meter (required) |
| is_primary | BOOLEAN | Default true; indicates the primary/main meter |
| added_date | DATE | Required; date this meter was linked to the agreement |
| removed_date | DATE | Null = currently linked; populated when meter is removed |
| created_at | TIMESTAMPTZ | |

**Relationships:**
- `serviceAgreement` → ServiceAgreement
- `meter` → Meter

**Constraint:** A meter can only be in one active ServiceAgreement per commodity at a time (enforced via `$transaction`).

## API Endpoints

All endpoints require JWT authentication with `utility_id` claim.

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/v1/service-agreements` | List service agreements (paginated, filterable) |
| POST | `/api/v1/service-agreements` | Create a new service agreement (with meter assignments) |
| GET | `/api/v1/service-agreements/:id` | Get agreement by ID (includes meters, premise, account) |
| PATCH | `/api/v1/service-agreements/:id` | Update agreement fields (status, rate, cycle, end date) |
| POST | `/api/v1/service-agreements/:id/meters` | Add a meter to an existing agreement |
| PATCH | `/api/v1/service-agreements/:id/meters/:samId` | Remove a meter from an agreement (sets `removed_date`) |

**Query parameters for `GET /service-agreements`:**

| Parameter | Type | Description |
|-----------|------|-------------|
| page | integer | Default 1 |
| limit | integer | Default 20, max 500 |
| sort | string | Default `createdAt` |
| order | `asc` \| `desc` | Default `desc` |
| accountId | UUID | Filter to agreements on a specific account |
| premiseId | UUID | Filter to agreements at a specific premise |
| status | AgreementStatus | Filter by status |

**Create request body:**

| Field | Required | Validation |
|-------|----------|------------|
| agreementNumber | Yes | min 1, max 50 chars |
| accountId | Yes | UUID |
| premiseId | Yes | UUID |
| commodityId | Yes | UUID |
| rateScheduleId | Yes | UUID |
| billingCycleId | Yes | UUID |
| startDate | Yes | ISO date string |
| endDate | No | ISO date string |
| status | No | Default PENDING |

**Meter assignment object (within `meters` array):**

| Field | Required | Validation |
|-------|----------|------------|
| meterId | Yes | UUID |
| isPrimary | No | boolean, default false |

**Update request body (PATCH):**

| Field | Allowed |
|-------|---------|
| rateScheduleId | Yes |
| billingCycleId | Yes |
| endDate | Yes |
| status | Yes (subject to transition rules) |
| readSequence | Yes |

Fields that cannot be changed after creation: `agreementNumber`, `accountId`, `premiseId`, `commodityId`. These are structural and changing them would represent a fundamentally different agreement.

## Business Rules

### Status Transitions

Strict linear progression — no skipping:

```
PENDING → ACTIVE → FINAL → CLOSED
```

Implemented in `packages/shared/src/validators/service-agreement.ts`:

```typescript
export const VALID_STATUS_TRANSITIONS: Record<AgreementStatus, AgreementStatus[]> = {
  PENDING: ["ACTIVE"],
  ACTIVE: ["FINAL"],
  FINAL: ["CLOSED"],
  CLOSED: [],
};
```

Attempting an invalid transition (e.g., `PENDING → FINAL` or `ACTIVE → CLOSED`) returns a validation error. `CLOSED` is terminal; no further transitions are permitted.

### Meter Assignment Uniqueness

When creating a ServiceAgreement, the provided meters are validated to ensure none are already active on another agreement for the same commodity. This check runs inside a `$transaction`. If a conflict is detected, the entire agreement creation rolls back with an error naming the conflicting meter.

### Meter-Premise Commodity Match

At creation time, every meter in the `meters` array must:
1. Have a `commodity_id` matching the agreement's `commodity_id`
2. Have a `premise_id` matching the agreement's `premise_id`
3. Have a `commodity_id` that exists in the premise's `commodity_ids` array

All three conditions ensure the meter physically makes sense for the agreement.

### At Least One Meter Required

The `createServiceAgreementSchema` requires `meters` array with `min(1)`. An agreement cannot exist without at least one meter assignment. This ensures every agreement can produce meter reads and billing.

### Effective Dating

`start_date` and `end_date` define the period of service. The rate schedule and billing cycle in effect at billing time are resolved from the agreement's rate/cycle FKs. Rate schedule version history (effective dates on RateSchedule) allows the system to bill at the correct rate for a given period even if the rate changes mid-cycle.

### Read Sequence

`read_sequence` is an integer that places this agreement in order within its billing cycle's meter reading route. Lower numbers are read first. This field supports route optimization for manual meter reading.

### Structural Immutability

The following fields cannot be changed after agreement creation:
- `agreementNumber` — permanent human-readable reference
- `accountId` — agreements belong to one account; to move service, close and re-create
- `premiseId` — agreements are at one location; to move service, close and re-create
- `commodityId` — an agreement is for one commodity; a separate agreement handles each commodity

### Soft Delete / Close

Agreements are never hard-deleted. The terminal state is `CLOSED` with `end_date` populated.

### Filters for List View

The GET list endpoint supports filtering by `accountId` and `premiseId` to support embedded list views within Account Detail and Premise Detail pages.

## UI Pages

| Page | Path | Features |
|------|------|----------|
| Agreements List | `/service-agreements` | Table with agreement number, account, premise, commodity, status, start date; filter by status |
| Agreement Detail | `/service-agreements/:id` | Tabs: Overview (inline editable fields + status transition buttons), Meters (assigned meters with add/remove actions), Audit (change history from AuditLog) |
| Agreements at Account | `/accounts/:id` (Agreements tab) | List of agreements for an account; status indicators |
| Agreements at Premise | `/premises/:id` (Agreements tab) | List of agreements at a premise; current vs historical; Add Agreement inline form |

**Overview tab fields displayed:** agreement number, status badge, account (linked), premise (linked), commodity, rate schedule (linked), billing cycle, start date, end date (if set), read sequence. All editable fields support inline editing.

**Status transition buttons:** Context-sensitive action buttons on the Overview tab:
- Agreement in `PENDING`: "Activate" button (transitions to ACTIVE)
- Agreement in `ACTIVE`: "Close" button (transitions to FINAL; confirmation dialog)
- Agreement in `FINAL`: "Finalize" button (transitions to CLOSED; confirmation dialog)

**Meters tab:** Displays all current and historical ServiceAgreementMeter records with `added_date` and `removed_date`. Add Meter button opens inline form to add a new meter assignment (validates commodity match and uniqueness). Remove button on active assignments sets `removed_date` via PATCH `/api/v1/service-agreements/:id/meters/:samId`.

**Audit tab:** Pulls from AuditLog filtered by `entity_type = 'ServiceAgreement'` and `entity_id`. Shows each change with actor, timestamp, and before/after state diff.

**Add Agreement inline form (Premise detail Agreements tab):** Creates a ServiceAgreement pre-filled with `premiseId`. Fields: agreement number, account (SearchableSelect), commodity (filtered to premise commodities), rate schedule, billing cycle, start date (DatePicker). Defaults status to PENDING.
